# LLM Side-Channel Topic Classification

Reproducing the topic classification attack from **"What Was Your Prompt? A Remote Keylogging Attack on AI Assistants"** (Weiss et al., USENIX Security 2024).

Week 9 experiment — ML Security Seminar.

## What this does

LLM responses are streamed token by token. Even over encrypted HTTPS, the size of each streamed chunk leaks the length of the token inside it. An eavesdropper who can't read the content can still see a sequence like `[3, 5, 12, 2, 8, ...]`.

This project shows that sequence alone is enough to classify the topic of the conversation, with no access to the actual text.

## Project structure

```
1_generate_data.py       Queries an LLM (streaming mode), records only chunk lengths + topic label
2_train_evaluate.py      Extracts statistical features, trains classifiers, evaluates
extract_size_pcap.py     Optional: extracts packet sizes from a PCAP capture (Bonus 1)
data/
  README.md              Data format notes
  generation_log.txt      Terminal output from running 1_generate_data.py
  example_pcap_output.txt Sample output from extract_size_pcap.py against real traffic
  responses.json          Generated by step 1
results/
  run1_overfitting/       First run — overfit (full-fit accuracy much higher than CV accuracy)
                          includes train_log.txt (terminal output) + plots
  run2_final/             Final run — fixed feature set, used for the presentation
                          includes train_log.txt (terminal output) + plots
```

## Setup

```bash
pip install -r requirements.txt
```

Get a free API key at [console.groq.com](https://console.groq.com), then either:
- set it as an environment variable: `export GROQ_API_KEY="your_key"`, or
- copy `.env.example` to `.env` and fill it in

We originally planned to use Gemini, but its free tier was cut to 20 requests/day during the project, which isn't enough for our 75 prompts. We switched to Llama 4 Scout via Groq (1,000 requests/day free, supports streaming).

## Running

**Step 1 — collect data** (~10 minutes, 75 API calls):
```bash
python 1_generate_data.py
```
Output: `data/responses.json`

**Step 2 — train and evaluate:**
```bash
python 2_train_evaluate.py
```
Output: confusion matrices, feature importance plots, and a comparison chart in `results/`

## Experiments

- **2 topics:** Poetry vs. Python code — the most structurally distinct pair, used as a sanity check
- **5 topics:** Poetry, Python code, medical advice, cooking, legal — the realistic, harder setting

## Results (final run — see `results/run2_final/`)

| Classifier          | 2-topic CV accuracy | 5-topic CV accuracy |
|----------------------|:---:|:---:|
| Logistic Regression | 83.3% | 44.0% |
| Random Forest       | 80.0% | 44.0% |
| Random baseline     | 50% | 20% |

Both classifiers land well above chance in both settings. Accuracy drops substantially from 2 to 5 topics, which is expected — the realistic open-ended classification problem is harder than the best-case sanity check.

Python code is the easiest topic to identify (near-perfect recall — its tokenizer pattern, with symbols and indentation, is very distinctive). Poetry is the hardest in the 5-topic setting, frequently confused with cooking and legal text.

The top feature in both experiments is `max` (maximum token length in the sequence), which is worth discussing as a limitation — see below.

### Note on `run1_overfitting/`

Our first run showed 100% full-fit accuracy on the 2-topic task, while cross-validation accuracy was only 83%. That gap is a classic overfitting signal — full-fit accuracy was being inflated by the model's exposure to the entire dataset during fitting. We kept this run in the repo because the overfitting diagnosis and fix are part of our methodology story (see presentation).

## Methodology notes

- **Train/test split is grouped by prompt index, not by response.** If the same prompt appeared in both train and test, the model could learn the prompt's structure rather than the topic's. Splitting by prompt index prevents this.
- **Features are statistics over the token-length sequence**, not the raw sequence: mean, std, max, min, median, sequence length, and a 5-bin normalized histogram. With ~75 samples, a model that consumes raw sequences (e.g. an LSTM) would overfit; hand-crafted features on top of a linear classifier are more appropriate here.
- **Tokenizer dependence:** the classifier is trained and evaluated only on Llama/Groq responses. A different LLM (e.g. GPT-4) uses a different tokenizer, so token-length distributions would differ. A real attacker would need to know or infer which LLM they're targeting.
- **Simulated, not sniffed:** we collect token lengths via the API's streaming mode rather than real packet capture. `extract_size_pcap.py` is a step toward validating this against real traffic (Bonus 1), but the chunk-to-packet mapping isn't one-to-one in practice.

## Limitations

- ~75 samples total — small, and per-fold accuracy has high variance (see per-fold scores printed by `2_train_evaluate.py`)
- Prompts were written by us (with LLM assistance for diversity); topic and prompt selection could bias results
- Only 5, deliberately distinct topics — harder, more similar topic pairs (e.g. medical vs. legal) would likely show lower accuracy than the 2-topic sanity check suggests
- Single LLM (Llama 4 Scout); doesn't necessarily generalize to other models' tokenizers
