# Week 9 – Side Channel Attack on LLM Systems

Reproducing the topic classification attack from:  
**"What Was Your Prompt? A Remote Keylogging Attack on AI Assistants"** – Weiss et al., USENIX Security 2024

## What This Does

LLM responses are streamed token by token. Even over encrypted traffic, the length of each token leaks as a packet size. This project shows that a sequence of token lengths alone — no text content — is enough to classify the topic of a conversation.

## Project Structure

```
week9_side_channel/
├── 1_generate_data.py      # Query Gemini in streaming mode, record token lengths
├── 2_train_evaluate.py     # Extract features, train classifier, evaluate
├── data/
│   └── responses.json      # Generated by step 1 (not committed to git)
├── results/                # Plots saved here by step 2
└── requirements.txt
```

## Setup

```bash
pip install -r requirements.txt
```

Get a free Gemini API key at https://aistudio.google.com  
Then open `1_generate_data.py` and replace `YOUR_GEMINI_API_KEY_HERE` with your key.

## Running

**Step 1 — collect data** (~2 minutes, makes ~75 API calls):
```bash
python 1_generate_data.py
```
Output: `data/responses.json`

**Step 2 — train and evaluate:**
```bash
python 2_train_evaluate.py
```
Output: confusion matrices and comparison plot in `results/`

## Experiments

- **Experiment A (2 topics):** Poetry vs. Python code — sanity check, maximally distinct classes
- **Experiment B (5 topics):** All 5 topics — more realistic, harder classification

Random baseline: 50% (2-topic), 20% (5-topic).

## Methodological Notes

- Token lengths are collected from streaming API chunks, not real network packets (simulated attack)
- Train/test split is done by prompt index, not by response, to prevent data leakage
- Prompts were manually reviewed for diversity to avoid topic-specific phrasing bias
- Classifier trained and tested entirely on Gemini responses — won't transfer to GPT-4 without retraining (different tokenizer)
